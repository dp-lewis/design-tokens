Design Tokens

One component, many brands — zero duplication

Interactive demo & discussion · 15 min

The problem

Same component, different brands

A story tile appears on every brand's homepage.
Each brand wants its own colours, typography, and spacing.

Common approach: duplicate the component per brand, or scatter overrides across stylesheets.

What goes wrong

  • Values drift between brands — "is that #0a1633 or #0a1733?"
  • Changing a colour means find-and-replace across files
  • Designers and developers have no shared vocabulary
  • Dark mode multiplies the problem

What are design tokens?

Named design decisions stored as data — not hard-coded values,
but meaningful names that everyone can share.

Primitive tokens

Raw values with no opinion about where they're used.
They're the palette — shared across every brand.

Primitive
color-navy-900  →  #0a1633
color-blue-600  →  #096dd2
font-playfair  →  "Playfair Display"
font-nunito  →  "Nunito"
spacing-sm  →  8px
spacing-xl  →  24px
These never change between brands. Every brand draws from the same set.

Semantic tokens

Describe what a value is for, not what it is.
Each brand maps these to different primitives.

Semantic
heading-color  →  color-navy-900
accent-color  →  color-blue-600
heading-font  →  font-playfair
body-font  →  font-nunito
space-loose  →  spacing-xl
This is where brand identity lives. A broadsheet points heading-font at a serif;
a tabloid points it at a sans-serif. Same token name, different value.

Component tokens

Scoped to a specific component.
They reference semantic tokens, adding one more layer of meaning.

Component
tile-headline-color  →  heading-color
tile-accent-color  →  accent-color
tile-headline-font  →  heading-font
tile-body-font  →  body-font
tile-padding  →  space-loose
A button, a card, a nav bar — each can have its own token set,
all pointing back to the same semantic layer.

What to notice

  • Brand switch — same HTML, completely different look
  • Layout switch — elements rearrange, tokens stay the same
  • Dark mode — each brand has its own dark palette
  • Token inspector — edit a primitive and see it ripple through

Where Figma fits in

The same idea, different tools

In Figma

  • Variables = primitive tokens
  • Modes = brand overrides
  • Auto layout = structure

In code

  • CSS custom properties = primitive tokens
  • Brand stylesheets = semantic mappings
  • CSS Grid / layout classes = structure
Designers and developers share the same token names, the same layers, the same mental model.

A shared language

When a designer says heading-color
and a developer writes var(--heading-color)
they mean the same thing.

Tokens bridge the gap between design and code.

What tokens give us

  • Consistency — one source of truth, not scattered values
  • Speed — rebrand by changing a file, not a codebase
  • Shared vocabulary — designers and developers speak the same language
  • Composability — appearance and structure are independent
