Design Tokens

One component, many brands — zero duplication

Interactive demo & discussion · 15 min

The problem

Same component, different brands

A story tile appears on every brand's homepage.
Each brand wants its own colours, typography, and spacing.

Common approach: maintain separate designs per brand, or copy-paste values and hope they stay in sync.

What goes wrong

  • You update the brand blue in one place — but it doesn't change everywhere
  • Design specs say 24px spacing, but what shipped is 16px
  • Designers and developers describe the same decision with different words
  • Add dark mode and you're maintaining twice as many values

What are design tokens?

Named design decisions stored as data — not hard-coded values,
but meaningful names that everyone can share.

Primitive tokens

Think of primitives as paint swatches on a wall — they exist before
anyone decides which room they go in. They're the raw palette, shared across every brand.

Primitive
color-navy-900  →  #0a1633
color-blue-600  →  #096dd2
font-playfair  →  "Playfair Display"
font-nunito  →  "Nunito"
spacing-sm  →  8px
spacing-xl  →  24px
These never change between brands. Every brand draws from the same set.

Semantic tokens

Every brand uses heading-font — they just point it at a different typeface.
Same token name, different value. That's where brand identity lives.

Semantic
heading-color  →  color-navy-900
accent-color  →  color-blue-600
heading-font  →  font-playfair
body-font  →  font-nunito
space-loose  →  spacing-xl
Semantic tokens describe what a value is for, not what it is. A broadsheet points heading-font at a serif; a tabloid points it at a sans-serif.

Component tokens

Why not use semantic tokens directly? Because a button and a story tile might both need a "colour" — but not the same colour. Component tokens let each component make its own choices from the shared palette.

Component
tile-headline-color  →  heading-color
tile-accent-color  →  accent-color
tile-headline-font  →  heading-font
tile-body-font  →  body-font
tile-padding  →  space-loose
A button, a card, a nav bar — each has its own token set, all pointing back to the same semantic layer.

Where Figma fits in

The same layers, different tools

In Figma

  • Variables hold primitives and semantics
  • Modes switch between brands — and between light/dark
  • Auto layout controls structure

In code

  • CSS custom properties define each layer
  • Brand stylesheets swap the semantic mappings
  • CSS Grid / layout classes control structure
The concepts are identical — only the tool changes. A token called heading-color means the same thing in a Figma variable as it does in a CSS file.

A shared language

When a designer says heading-color
and a developer writes var(--heading-color)
they mean the same thing.

Tokens bridge the gap between design and code.

We've prepared a Figma document that walks through this in detail — variables, modes, and how they map to what you've seen in the demo.

What tokens give us

  • Consistency — one source of truth, not scattered values
  • Speed — rebrand by changing a file, not a codebase
  • Shared vocabulary — designers and developers speak the same language
  • Composability — appearance and structure are independent
